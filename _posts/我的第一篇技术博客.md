## 一、OSI七层模型

### 1.1 什么是OSI模型？

OSI（Open Systems Interconnection）模型是国际标准化组织（ISO）提出的网络通信参考模型，将网络通信过程分为七个层次，每层实现特定的功能。

### 1.2 七层详解

|层级	   |名称	       |功能	                   |协议/设备	                 |网安关注点                     |
|--------|------------|--------------------------|-----------------------------|------------------------------|
|第7层	|应用层	    |为应用程序提供网络服务接口	 |HTTP、HTTPS、FTP、DNS、SMTP  |应用层攻击（SQL注入、XSS、CSRF）|
|第6层	|表示层	    |数据格式转换、加密解密	     |SSL/TLS、JPEG、MPEG	       |中间人攻击、证书伪造           |
|第5层	|会话层	    |建立、管理、终止会话        |RPC、NFS、SQL会话	           |会话劫持                      |
|第4层	|传输层	    |端到端的可靠数据传输	     |TCP、UDP	                   |端口扫描、SYN Flood攻击       |
|第3层	|网络层	    |路由选择、逻辑地址（IP）	 |IP、ICMP、ARP、OSPF	       |IP欺骗、路由劫持               |
|第2层	|数据链路层	|物理地址（MAC）、帧传输	 |Ethernet、MAC地址	           |ARP欺骗、交换机攻击             |
|第1层	|物理层	    |物理连接、比特流传输	     |网线、光纤、集线器	        |物理窃听、信号干扰              |

├─────────────────────────────────────────┤
│ 传输层：TCP头部（端口号）+ 应用层数据 │

### 1.4 网安视角的七层模型

**第7层（应用层）：攻击高发区**
- SQL注入、XSS、CSRF
- 文件上传漏洞
- 逻辑漏洞

**第6层（表示层）：加密与解密**
- HTTPS中间人攻击
- SSL/TLS降级攻击
- 证书伪造

**第4层（传输层）：连接管理**
- TCP SYN Flood攻击
- 端口扫描
- 会话劫持

**第3层（网络层）：路由与寻址**
- IP欺骗
- 路由劫持
- ICMP隧道攻击

**第2层（数据链路层）：物理地址**
- ARP欺骗
- MAC泛洪攻击
- VLAN跳转攻击

## 二、TCP/IP协议栈

### 2.1 OSI与TCP/IP的对应关系

虽然OSI是理论模型，但实际网络使用的是TCP/IP协议栈。两者对应关系如下：

|OSI七层模型	       |TCP/IP四层模型	|对应协议                 |
|-----------------------|---------------|-------------------------|
|应用层、表示层、会话层	|应用层          |HTTP、FTP、DNS、SMTP、SSH|
|传输层	               |传输层          |TCP、UDP                |
|网络层	               |网际层          |IP、ICMP、ARP           |
|数据链路层、物理层	    |网络接口层      |Ethernet、Wi-Fi、PPP     |

### 2.2 为什么TCP/IP只有四层？

**OSI模型**是理论模型，层次划分细致，但过于复杂，实际应用中效率较低。

**TCP/IP协议栈**是实际使用的协议族，将OSI的上三层合并为应用层，下两层合并为网络接口层，更加实用高效。

### 2.3 TCP/IP协议栈结构

-┌────────────────────────────────────┐
-│ 应用层 │
-│ HTTP、HTTPS、FTP、DNS、SMTP │
-├────────────────────────────────────┤
-│ 传输层 │
-│ TCP（可靠传输）、UDP（快速传输） │
-├────────────────────────────────────┤
-│ 网际层 │
-│ IP（路由）、ICMP（ping）、ARP │
-├────────────────────────────────────┤
-│ 网络接口层 │
-│ Ethernet、Wi-Fi、PPP │
-└────────────────────────────────────┘

## 三、TCP协议详解

### 3.1 什么是TCP？

TCP（Transmission Control Protocol，传输控制协议）是传输层的核心协议，提供：
- **可靠传输**：通过序列号、确认号、重传机制确保数据完整
- **面向连接**：传输前需要建立连接（三次握手）
- **全双工**：双向可以同时传输数据
- **流量控制**：通过滑动窗口控制数据传输速率
- **拥塞控制**：通过慢启动、拥塞避免算法避免网络拥塞

### 3.2 TCP头部结构

**关键字段说明：** 

| 字段                  | 位数  | 说明        | 网安关注点            |
|-----------------------|------|------------|-----------------------|
| Source Port           | 16位 | 源端口号    | 端口扫描识别服务        |
| Destination Port      | 16位 | 目的端口号  | 端口扫描识别服务        |
| Sequence Number       | 32位 | 序列号      | 序列号攻击             |
| Acknowledgment Number | 32位 | 确认号      | 确认号伪造             |
| Flags                 | 6位  | 标志位      | SYN、ACK、FIN、RST攻击 |
| Window                | 16位 | 滑动窗口大小 | 流量控制攻击           |

**标志位详解：** 

| 标志| 名称            | 说明               | 攻击场景             |
|-----|----------------|--------------------|---------------------|
| SYN | Synchronize    | 同步序列号，建立连接 | SYN Flood           |
| ACK | Acknowledgment | 确认号有效          | ACK Flood           |
| FIN | Finish         | 关闭连接            | FIN扫描              |
| RST | Reset          | 复位连接            | RST攻击、强制中断连接 |
| PSH | Push           | 立即推送数据         |                     |
| URG | Urgent         | 紧急指针有效         | URG攻击              |

### 3.3 TCP vs UDP

| 特性     | TCP                     | UDP                |
|---------|--------------------------|--------------------|
| 连接     | 面向连接                 | 无连接              |
| 可靠性   | 可靠传输                 | 不可靠传输          |
| 传输效率 | 较低（三次握手、确认重传） | 高（无连接、无确认） |
| 顺序     | 保证数据顺序             | 不保证顺序           |
| 流量控制 | 有（滑动窗口）            | 无                  |
| 拥塞控制 | 有（慢启动、拥塞避免）     | 无                  |
| 应用场景 | HTTP、FTP、SMTP、SSH      | DNS、视频流、游戏   |

## 四、TCP三次握手（建立连接）

### 4.1 为什么需要三次握手？

**核心原因：** 
1. **防止已失效的连接请求突然传送到服务端**
   - 客户端发送的第一个SYN包因网络拥塞滞留
   - 客户端超时重发第二个SYN包，建立连接
   - 连接结束后，滞留的SYN包突然到达服务端
   - 服务端误以为是新的连接请求，发送SYN+ACK并建立连接
   - 客户端丢弃服务端的SYN+ACK（因为连接已关闭）
   - **问题：服务端一直等待，浪费资源**

2. **确认双方都能收发数据**
   - 两次握手只能确认"服务端能收到客户端的数据"
   - 三次握手确认"客户端也能收到服务端的数据"
   - 确保双向通信能力

3. **同步初始序列号**
   - 双方交换各自的初始序列号（ISN）
   - 为后续可靠传输奠定基础

### 4.2 三次握手详细过程

-客户端                              服务端
-   │                                  │
-   │  ① SYN包（seq=x）                 │
-   │────────────────────────────────> │
-   │   (请求建立连接)                  │
-   │   [SYN=1, seq=x, ack=0]          │
-   │  ② SYN+ACK包（seq=y, ack=x+1）    │
-   │<──────────────────────────────── │
-   │   (确认收到请求 + 请求建立连接)    │
-   │   [SYN=1, ACK=1, seq=y, ack=x+1] │
-   │  ③ ACK包（seq=x+1, ack=y+1）      │
-   │────────────────────────────────> │
-   │   (确认收到对方的请求)             │
-   │   [ACK=1, seq=x+1, ack=y+1]       │
-   │         连接建立成功               │


### 4.3 三次握手包详解

**第一次握手（客户端 → 服务端）：** 
-标志位：SYN=1, ACK=0
-序列号：seq=x（客户端初始序列号，随机生成）
-确认号：ack=0（首次握手，无需确认）
-含义：客户端请求建立连接，初始序列号为x
-状态转换：客户端 → SYN_SENT

**第二次握手（服务端 → 客户端）：** 
-标志位：SYN=1, ACK=1
-序列号：seq=y（服务端初始序列号，随机生成）
-确认号：ack=x+1（确认收到客户端的seq=x，下一个期望收到x+1）
-含义：服务端确认收到连接请求，同时请求建立连接
**状态转换：**
-服务端 → SYN_RCVD
-客户端 → ESTABLISHED（收到SYN+ACK后）

**第三次握手（客户端 → 服务端）：** 
-标志位：ACK=1
-序列号：seq=x+1（客户端的序列号，从x+1开始）
-确认号：ack=y+1（确认收到服务端的seq=y，下一个期望收到y+1）
-含义：客户端确认收到服务端的连接请求
**状态转换：**
-客户端 → ESTABLISHED
-服务端 → ESTABLISHED（收到ACK后）

### 4.4 为什么不是两次握手？

如果只有两次握手（客户端SYN → 服务端SYN+ACK），会出现以下问题：

**问题1：失效的连接请求**
时间线：
-T1: 客户端发送SYN包A，因网络拥塞滞留
-T2: 客户端超时重发SYN包B，服务端收到，建立连接
-T3: 连接结束，双方关闭连接
-T4: 滞留的SYN包A突然到达服务端
-T5: 服务端误以为是新的连接请求，发送SYN+ACK并建立连接
-T6: 客户端丢弃服务端的SYN+ACK（因为连接已关闭）
-T7: 服务端一直等待，浪费资源

**问题2：双方收发能力验证**
- 两次握手只能确认"服务端能收到客户端的数据"
- 无法确认"客户端能收到服务端的数据"
- 三次握手确保双向通信能力

### 4.5 三次握手Wireshark抓包示例

|No. |Time| Source |Destination |Protocol |Info|
|----|----|--------------|-----------|----|----------------------------------|
|1 |0.000000| 192.168.1.100| 192.168.1.200| TCP |60 → 443 [SYN] Seq=0 Len=0 Win=64240|
|2 |0.001234 |192.168.1.200 |192.168.1.100 |TCP |443 → 60 [SYN, ACK] Seq=0 Ack=1 Win=65535|
|3 |0.002567 |192.168.1.100 |192.168.1.200 |TCP| 60 → 443 [ACK] Seq=1 Ack=1 Win=64240|

**解读：** 
1. **客户端**发送SYN包，seq=0（初始序列号）
2. **服务端**回复SYN+ACK，seq=0（自己的初始序列号），ack=1（确认收到seq=0）
3. **客户端**回复ACK，seq=1（从1开始），ack=1（确认收到seq=0）

## 五、TCP四次挥手（终止连接）

### 5.1 为什么需要四次挥手？

**核心原因：TCP是全双工协议**

全双工意味着：
- 数据可以双向同时传输
- 连接关闭时，两个方向的数据流可以独立关闭

**场景示例：** 
-客户端：我已经不发数据了，可以关闭了（FIN）
-服务端：好的，我知道你不发了（ACK），但我还有10MB数据要发给你
-[服务端发送10MB数据...]
-服务端：数据发完了，我也可以关闭了（FIN）
-客户端：好的，我知道了（ACK）

### 5.2 四次挥手详细过程
-客户端                              服务端
-   │                                  │
-   │  ① FIN包（seq=u）                │
-   │────────────────────────────────> │
-   │   (客户端请求关闭连接)            │
-  │   [FIN=1, seq=u]                │
-   │  ② ACK包（ack=u+1）              │
-   │<──────────────────────────────── │
-   │   (服务端确认收到关闭请求)         │
-   │    [ACK=1, seq=v, ack=u+1]       │
-   │  ③ FIN包（seq=w）                 │
-   │<──────────────────────────────── │
-   │   (服务端也请求关闭连接)           │
-   │   [FIN=1, seq=w, ack=u+1]        │
-   │  ④ ACK包（ack=w+1）               │
-   │────────────────────────────────> │
-   │   (客户端确认收到关闭请求)         │
-   │   [ACK=1, seq=u+1, ack=w+1]      │
-   │         连接完全关闭              │

### 5.3 四次挥手包详解

**第一次挥手（客户端 → 服务端）：** 
-标志位：FIN=1
-序列号：seq=u（客户端当前的序列号）
-确认号：ack=（可能包含，如果有数据传输）
-含义：客户端请求关闭连接，不再发送数据
-状态转换：客户端 → FIN_WAIT_1

**第二次挥手（服务端 → 客户端）：** 
-标志位：ACK=1
-序列号：seq=v（服务端当前的序列号）
-确认号：ack=u+1（确认收到客户端的seq=u，下一个期望收到u+1）
-含义：服务端确认收到关闭请求，但可能还有数据要发送
**状态转换：**
-客户端 → FIN_WAIT_2
-服务端 → CLOSE_WAIT

**第三次挥手（服务端 → 客户端）：** 
-标志位：FIN=1
-序列号：seq=w（服务端的序列号，可能等于v，也可能因发送数据而变化）
-确认号：ack=u+1（仍然是确认收到客户端的FIN）
-含义：服务端的数据都发完了，现在可以关闭了
**状态转换：**
-服务端 → LAST_ACK

**第四次挥手（客户端 → 服务端）：** 
-标志位：ACK=1
-序列号：seq=u+1（客户端的序列号）
-确认号：ack=w+1（确认收到服务端的seq=w，下一个期望收到w+1）
-含义：客户端确认收到服务端的关闭请求，连接彻底关闭
**状态转换：**
-客户端 → TIME_WAIT（等待2MSL）
-服务端 → CLOSED
-客户端 → CLOSED（2MSL后）

### 5.4 为什么第二次和第三次挥手不能合并？

**原因：TCP全双工协议的特性**

三次握手的第二次握手（SYN+ACK）可以合并，因为服务端在确认客户端SYN的同时，也可以发送自己的SYN请求。

但四次挥手的第二次（ACK）和第三次（FIN）不能合并，因为：
1. 服务端收到客户端FIN后，确认收到（ACK）
2. 但服务端可能还有数据要发送给客户端
3. 数据发送完后，服务端才能发送FIN请求关闭连接
4. 这是两个独立的动作，不能合并

**示意图：** 
**不合并的情况（正常）：**
-客户端 → FIN
-服务端 → ACK（确认收到FIN）
-[服务端发送数据...]
-服务端 → FIN（数据发完了）
-客户端 → ACK
**合并的情况（错误）：**
-客户端 → FIN
-服务端 → ACK+FIN（问题：服务端还有数据没发完）
-[数据丢失！]

### 5.5 TIME_WAIT状态

**第四次挥手后，客户端进入TIME_WAIT状态，持续2MSL（Maximum Segment Lifetime，最大报文生存时间）。** 

**为什么需要TIME_WAIT？** 

1. **确保最后一个ACK能够到达服务端**
   - 如果ACK丢失，服务端会重传FIN
   - 客户端在TIME_WAIT状态，可以重传ACK

2. **确保旧连接的数据包在网络中消失**
   - 2MSL后，网络中属于该连接的所有数据包都会消失
   - 防止新连接收到旧连接的数据包

**2MSL是多久？** 
- MSL（Maximum Segment Lifetime）通常为30秒到2分钟
- 2MSL即1到4分钟

### 5.6 四次挥手Wireshark抓包示例
|No. |Time     |Source        |Destination   |Protocol  |Info|
|----|---------|--------------|--------------|----------|----------------------------------------------|
|1   |0.000000 |192.168.1.100 |192.168.1.200 |TCP       |60 → 443 [FIN, ACK] Seq=1234 Ack=5678 Win=64240|
|2   |0.001234 |192.168.1.200 |192.168.1.100 |TCP       |443 → 60 [ACK] Seq=5678 Ack=1235 Win=65535|
|3   |0.050000 |192.168.1.200 |192.168.1.100 |TCP       |443 → 60 [FIN, ACK] Seq=5678 Ack=1235 Win=65535|
|4   |0.051234 |192.168.1.100 |192.168.1.200 |TCP       |60 → 443 [ACK] Seq=1235 Ack=5679 Win=64240|

**解读：** 
1. **客户端**发送FIN+ACK，seq=1234（当前序列号），ack=5678（确认收到服务端的5678）
2. **服务端**回复ACK，seq=5678（当前序列号），ack=1235（确认收到客户端的1234）
3. **服务端**发送FIN+ACK，seq=5678（当前序列号），ack=1235（确认）
4. **客户端**回复ACK，seq=1235（从1235开始），ack=5679（确认收到5678）

## 六、网安重点：TCP安全攻击与防御

### 6.1 SYN Flood攻击

**攻击原理：** 
攻击者发送大量SYN包但不回复第三次握手的ACK，使服务端维护大量半连接状态，耗尽资源。

**攻击过程：** 
-攻击者                           服务端
-│                                  │
-│ ① SYN包1（seq=随机1）             │
-│────────────────────────────────> │
-│ （进入半连接队列）                 │
-│ ② SYN包2（seq=随机2）             │
-│────────────────────────────────> │
-│ （继续占用资源）                  │
-│ ③ SYN包3（seq=随机3）             │
-│────────────────────────────────> │
-│ （半连接队列满）                   │
-│                                  │
-│ [攻击者不回复第三次握手ACK]        │
-│ [服务端持续等待...]                │
-│ [正常用户无法建立连接]             │

**攻击特征：** 
- 大量SYN包，但无ACK回复
- 半连接队列（SYN_RCVD状态）激增
- 服务端响应缓慢或无响应

**防御措施：** 

| 防御措施 | 原理 | 实施难度 |
|---------|------|---------|
| **SYN Cookies** | 服务端不保存半连接状态，根据SYN包计算一个Cookie | 中等 |
| **SYN Cache** | 缓存SYN请求，限制半连接队列大小 | 简单 |
| **防火墙/IPS** | 检测异常SYN流量并拦截 | 简单 |
| **调整TCP参数** | 增加半连接队列、缩短超时时间 | 简单 |
| **DDoS防护服务** | 云服务商提供的DDoS防护 | 简单 |

**Linux系统防御配置：** 
```bash
# 启用SYN Cookies
echo 1 > /proc/sys/net/ipv4/tcp_syncookies

# 增加半连接队列长度
echo 2048 > /proc/sys/net/ipv4/tcp_max_syn_backlog

# 缩短SYN超时时间
echo 30 > /proc/sys/net/ipv4/tcp_synack_retries

# 启用TCP时间戳
echo 1 > /proc/sys/net/ipv4/tcp_timestamps

```

### 6.2 RST攻击
**攻击原理：**
攻击者伪造RST包（Reset，复位标志），强制中断TCP连接。
**攻击条件：**
攻击者需要知道客户端和服务端的IP、端口
攻击者需要猜测正确的序列号或确认号
**攻击过程：**
**正常连接：**
-  客户端                          服务端
-   │                                │
-   │  数据传输中...                  │
-   │  ←───────────────────────────> │
-   │  ←───────────────────────────> │
-   │                                │

**RST攻击：**
-攻击者
-  │                                │
-  │  伪造RST包                      │
-  │─────────────────────────────>  │
-  │                                │
-  │  [服务端收到RST，强制关闭连接]   │
-  │  [正常数据传输中断]              │

**攻击特征：**
-正常的TCP连接突然中断
-Wireshark抓包中发现异常的RST包
-RST包的序列号在合法窗口内

 **防御措施：**
|防御措施	 |原理	                          |实施难度|
|------------|--------------------------------|--------|
|加密序列号	 |使用TLS/SSL，攻击者无法猜测序列号 |简单    |
|序列号随机化|使攻击者难以猜测序列号	        |简单    |
|网络层安全	 |IPSec防止IP欺骗	               |困难    |
|RST包过滤	|防火墙过滤异常RST包	           |中等    |
|序列号保护	 |TCP序列号随机化（ISN随机化）	    |简单    |

### 6.3 TCP会话劫持
**攻击原理：**
攻击者通过猜测正确的序列号，插入到已建立的TCP连接中，劫持会话。
**正常连接：**
-客户端                          服务端
-   │                                │
-   │  SYN → SYN+ACK → ACK           │
-   │  连接建立                       │
-   │  ←───────────────────────────> │
-   │  ←───────────────────────────> │
-   │                                │

**会话劫持：**
-攻击者
-   │                                │
-   │  监听网络流量                   │
-   │  猜测正确的序列号               │
-   │                                │
-   │  伪造RST包给客户端              │
-   │─────────────────────────────>  │
-   │  [客户端连接中断]               │
-   │                                │
-   │  伪造数据包给服务端             │
-   │─────────────────────────────>  │
-   │  [服务端误以为是客户端发来的]    │
-   
**防御措施：**
|防御措施	   |原理                         |
|-----------|-----------------------------|
|加密传输	   |使用TLS/SSL，攻击者无法篡改数据|
|序列号随机化  |增加猜测难度                  |
|IPSec	      |提供网络层安全                |
|交换机端口安全|防止ARP欺骗                   |

### 6.4 端口扫描

**扫描原理：**
攻击者通过发送不同标志位的TCP包，探测目标主机的开放端口和服务。

**常见扫描类型：**
|扫描类型	      |原理                                |检测方法          |
|---------------|-------------------------------------|-----------------|
|TCP Connect扫描 |完整的三次握手，如果端口开放，连接成功  |检测大量SYN连接   |
|SYN扫描（半开）  |只发送SYN，根据SYN+ACK判断端口开放	  |检测大量SYN但无ACK|
|FIN扫描	     |发送FIN，根据RST或无响应判断	         |检测异常FIN包     |
|NULL扫描	     |发送无标志位的包，根据响应判断	      |检测异常TCP包     |
|XMAS扫描	     |发送所有标志位（FIN、PSH、URG）	     |检测异常TCP包      |

**防御措施：**
|防御措施	  |原理         |
|----------|--------------|
|防火墙	  |过滤异常TCP包  |
|IPS/IDS   |检测扫描行为   |
|端口隐藏	  |不回应扫描请求|
|日志监控 	|监控异常连接  |

## 7.3 检测SYN Flood攻击
**步骤：**
-在Wireshark过滤器中输入：tcp.flags.syn == 1 && tcp.flags.ack == 0
-观察是否有大量SYN包但无ACK回复
-查看源IP和目标端口

**预期结果:**
|No. |Time        |Source       |Destination    |Protocol  |Info|
|----|------------|-------------|---------------|----------|----------------------|
|1   |0.000000    |10.0.0.100   |192.168.1.200  |TCP       |12345 → 80 [SYN] Seq=0|
|2   |0.000010    |10.0.0.100   |192.168.1.200  |TCP       |12346 → 80 [SYN] Seq=0|
|3   |0.000020    |10.0.0.100   |192.168.1.200  |TCP       |12347 → 80 [SYN] Seq=0|
|4   |0.000030    |10.0.0.100   |192.168.1.200  |TCP       |12348 → 80 [SYN] Seq=0|
|5   |0.000040    |10.0.0.100   |192.168.1.200  |TCP       |12349 → 80 [SYN] Seq=0|
...
[大量SYN包，但无ACK回复]

**分析方法：**
-查看是否来自同一源IP（攻击源）

-检查是否有对应的ACK回复
-如果无ACK回复且数量巨大，可能是SYN Flood攻击

⏳ 疑问1：三次握手中的seq和ack为什么是x+1而不是x？

## 十二、总结
-核心要点（背诵版）
### OSI七层模型：
-物理层：比特流传输
-数据链路层：MAC地址、帧传输
-网络层：IP地址、路由选择
-传输层：端到端可靠传输（TCP/UDP）
-会话层：会话管理
-表示层：数据格式转换、加密
-应用层：应用层协议（HTTP、FTP、DNS）
### TCP三次握手：
-客户端SYN → 服务端
-服务端SYN+ACK → 客户端
-客户端ACK → 服务端
### TCP四次挥手：
-客户端FIN → 服务端
-服务端ACK → 客户端
-服务端FIN → 客户端
-客户端ACK → 服务端
### 网安重点：
-SYN Flood攻击：大量SYN但不ACK，耗尽服务端资源
-防御措施：SYN Cookies、SYN Cache、防火墙
-RST攻击：伪造RST包，强制中断连接
-防御措施：TLS/SSL、序列号随机化
